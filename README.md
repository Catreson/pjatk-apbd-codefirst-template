# LibraryShop - ASP.NET Core + EF Core Code-First Template

A complete, working example of an ASP.NET Core Web API using **Entity Framework Core (Code-First)**.
Everything in this repo is a real, runnable project — not pseudocode.

Domain: a bookshop where customers can place orders for books.

## Database Setup (LocalDB)

LocalDB is a lightweight SQL Server that ships with Visual Studio — no installation needed beyond VS.

### Step 1 — Verify LocalDB is available

Open a terminal and run:

```bash
sqllocaldb info
```

You should see `MSSQLLocalDB` in the list. If the command is not found, open **Visual Studio Installer**, go to **Individual components**, and install **SQL Server Express LocalDB**.

### Step 2 — Start the LocalDB instance (if not already running)

```bash
sqllocaldb start MSSQLLocalDB
```

LocalDB starts automatically when a connection is made, so this step is usually not needed.

### Step 3 — Connection String

The connection string in `LibraryShop/appsettings.json` is already configured for LocalDB:

```json
{
  "ConnectionStrings": {
    "Default": "Data Source=(localdb)\\MSSQLLocalDB;Initial Catalog=LibraryShop;Integrated Security=True;Connect Timeout=30;"
  }
}
```

The database `LibraryShop` will be created automatically when you run `dotnet ef database update`.

### Viewing the database

You can browse the LocalDB database in:
- **Visual Studio** → View → SQL Server Object Explorer → `(localdb)\MSSQLLocalDB` → Databases → LibraryShop
- **JetBrains Rider** → Database panel → `+` → Data Source → SQL Server → Server: `(localdb)\MSSQLLocalDB`

## Running the Project

### Step 1 — Restore NuGet packages

```bash
cd LibraryShop/LibraryShop
dotnet restore
```

### Step 2 — Apply EF Core migrations (creates the database + tables + seed data)

```bash
dotnet ef database update
```

This command:
1. Connects to SQL Server
2. Creates the `LibraryShop` database (if it doesn't exist)
3. Creates all tables (Author, Book, Category, etc.)
4. Inserts the seed data (4 authors, 8 books, 3 customers, etc.)

### Step 3 — Run the API

```bash
dotnet run
```

The output will show something like:

```
Now listening on: http://localhost:5277
Now listening on: https://localhost:7277
```

### Step 4 — Open Swagger UI

Open your browser and go to:
```
http://localhost:5277
```

You'll see the interactive Swagger UI where you can test all endpoints.

Alternatively, use `LibraryShop.http` in VS Code with the REST Client extension.

## EF Core Migration Commands

These commands must be run from the project directory: `LibraryShop/LibraryShop`

### Create a new migration (after you change any Model or Configuration)

```bash
dotnet ef migrations add <MigrationName>
```

Example:
```bash
dotnet ef migrations add AddBookISBN
```

This generates a new file in the `Migrations/` folder.
**Never edit migration files by hand.**

### Apply pending migrations to the database

```bash
dotnet ef database update
```

### Roll back to a specific migration

```bash
dotnet ef database update <MigrationName>
```

### Remove the LAST migration (only if not yet applied to the database)

```bash
dotnet ef migrations remove
```

### See all migrations and their status

```bash
dotnet ef migrations list
```

### Drop the entire database and recreate from scratch

```bash
dotnet ef database drop --force
dotnet ef database update
```

### Books

#### `GET /api/books` — List all books

Optional query parameters for filtering:

| Parameter | Type | Example | Description |
|-----------|------|---------|-------------|
| `authorLastName` | string | `Tolkien` | Filter by author surname (partial match) |
| `minPrice` | decimal | `20` | Only books priced >= this value |
| `maxPrice` | decimal | `40` | Only books priced <= this value |
| `category` | string | `Fantasy` | Only books in this category |
| `inStock` | bool | `true` | Only books with Stock > 0 |

## LINQ Examples Explained

LINQ (Language Integrated Query) lets you query collections and databases using C# syntax.
All LINQ operations on `DbSet<T>` are translated to SQL by EF Core.

### Example 1 — Get all books, project to DTO

```csharp
// IQueryable<Book> - no SQL yet
var query = _db.Books
    .Include(b => b.Author)               // JOIN Author
    .Include(b => b.BookCategories)       // JOIN BookCategory
        .ThenInclude(bc => bc.Category)   // JOIN Category (through BookCategory)
    .AsQueryable();

// Add a WHERE clause - still no SQL
query = query.Where(b => b.Price >= 20m);

// THIS line sends the SELECT to the database
var results = await query
    .OrderBy(b => b.Title)
    .Select(b => new BookListItemDto        // SELECT only what we need
    {
        Id         = b.Id,
        Title      = b.Title,
        AuthorName = b.Author.FirstName + " " + b.Author.LastName,
        Categories = b.BookCategories.Select(bc => bc.Category.Name).ToList()
    })
    .ToListAsync();
```

**Equivalent SQL:**
```sql
SELECT b.Id, b.Title, a.FirstName + ' ' + a.LastName AS AuthorName
FROM Book b
INNER JOIN Author a ON b.AuthorId = a.Id
WHERE b.Price >= 20
ORDER BY b.Title
```

---

### Example 2 — Conditional .Where() filters (query parameters)

```csharp
var query = _db.Books.AsQueryable();

// Each if block adds an AND condition to the SQL
if (!string.IsNullOrWhiteSpace(authorLastName))
    query = query.Where(b => b.Author.LastName.Contains(authorLastName));
//                                                      ^^^^^^^^^^^^^^^^^
//                                                      translates to SQL LIKE '%Tolkien%'

if (minPrice.HasValue)
    query = query.Where(b => b.Price >= minPrice.Value);

if (inStock == true)
    query = query.Where(b => b.Stock > 0);

// SQL is built up progressively - nothing hits the DB until ToListAsync()
var books = await query.ToListAsync();
```

---

### Example 3 — .FirstOrDefaultAsync() vs .ToListAsync()

```csharp
// Returns ONE item or null - adds "TOP 1" (or equivalent) to SQL
var book = await _db.Books.FirstOrDefaultAsync(b => b.Id == id);

// Returns ALL matching rows as a list
var books = await _db.Books.Where(b => b.AuthorId == 1).ToListAsync();

// Returns true/false - uses SQL EXISTS
var exists = await _db.Authors.AnyAsync(a => a.Id == authorId);
```

---

### Example 4 — .Select() for projection (DO THIS instead of Include + manual mapping)

```csharp
// BAD: loads the entire Order entity, then Author, then all OrderItems, then all Books
var order = await _db.Orders
    .Include(o => o.Customer)
    .Include(o => o.OrderItems).ThenInclude(oi => oi.Book)
    .FirstOrDefaultAsync(o => o.Id == id);
// Then manually map to DTO...

// GOOD: project directly - only fetches the columns we actually need
var dto = await _db.Orders
    .Select(o => new OrderDetailDto
    {
        Id        = o.Id,
        CreatedAt = o.CreatedAt,
        Customer  = new CustomerSummaryDto
        {
            FirstName = o.Customer.FirstName,   // EF Core auto-joins Customer
            LastName  = o.Customer.LastName,
        },
        Items = o.OrderItems.Select(oi => new OrderItemDto
        {
            BookTitle = oi.Book.Title,           // EF Core auto-joins Book
            Quantity  = oi.Quantity,
        }).ToList()
    })
    .FirstOrDefaultAsync(o => o.Id == id);     // filter goes at the end
```

---

### Example 5 — Aggregates in LINQ (.Sum, .Count, .Any)

```csharp
// Sum of (quantity * price) per order
decimal total = order.OrderItems.Sum(oi => oi.Quantity * oi.UnitPrice);

// Count of order lines
int lineCount = order.OrderItems.Count();

// Does any category match?
bool hasFantasy = book.BookCategories.Any(bc => bc.Category.Name == "Fantasy");

// These also work in .Select() projections sent directly to SQL:
.Select(c => new CustomerDetailDto
{
    Orders = c.Orders.Select(o => new CustomerOrderSummaryDto
    {
        TotalPrice = o.OrderItems.Sum(oi => oi.Quantity * oi.UnitPrice), // SQL SUM()
        ItemCount  = o.OrderItems.Count(),                                // SQL COUNT()
    }).ToList()
})
```

---

### Example 6 — .Include() and .ThenInclude() for loading related data

```csharp
// Load Book + Author + BookCategories + the Category of each BookCategory
var book = await _db.Books
    .Include(b => b.Author)                 // 1 level deep: Author
    .Include(b => b.BookCategories)         // 1 level deep: BookCategories collection
        .ThenInclude(bc => bc.Category)     // 2 levels deep: Category inside each BookCategory
    .FirstOrDefaultAsync(b => b.Id == id);
```

> **Note:** When you use `.Select()` for projection, EF Core handles joins automatically
> and you often don't need explicit `.Include()` calls.

### Example 7 — .OrderBy() and .OrderByDescending()

```csharp
// Sort books alphabetically by title
var books = await _db.Books.OrderBy(b => b.Title).ToListAsync();

// Sort by price descending (most expensive first)
var expensive = await _db.Books.OrderByDescending(b => b.Price).ToListAsync();

// Sort by author name, then by title
var sorted = await _db.Books
    .OrderBy(b => b.Author.LastName)
    .ThenBy(b => b.Title)
    .ToListAsync();

// Most recent orders first
var orders = await _db.Orders.OrderByDescending(o => o.CreatedAt).ToListAsync();
```

## Key Concepts Explained

### Code-First vs Database-First

- **Code-First:** You write C# classes (Models), then EF Core generates the database from them. This is what we do here.
- **Database-First:** You design the database first, then EF Core generates C# classes from it.

### IEntityTypeConfiguration\<T\>

Instead of putting all table configuration in `OnModelCreating` (which gets huge), we split it into one file per model:

```csharp
// Configurations/BookConfiguration.cs
public class BookConfiguration : IEntityTypeConfiguration<Book>
{
    public void Configure(EntityTypeBuilder<Book> builder)
    {
        builder.HasKey(b => b.Id);
        builder.Property(b => b.Title).HasMaxLength(200).IsRequired();
        builder.HasData(/* seed data */);
    }
}
```

Then in `AppDbContext`:

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    // This ONE line finds ALL IEntityTypeConfiguration<T> classes in the project
    modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
}
```

### Dependency Injection (DI)

Instead of creating objects manually with `new`, ASP.NET Core creates them for you:

```csharp
// Program.cs - register your service
builder.Services.AddScoped<IBookService, BookService>();

// Controller - ASP.NET Core automatically creates BookService and passes it here
public class BooksController : ControllerBase
{
    private readonly IBookService _bookService;

    public BooksController(IBookService bookService) // <-- injected automatically
    {
        _bookService = bookService;
    }
}
```

### Scoped Lifetime (AddScoped)

- `AddScoped` — one instance per HTTP request. Use this for everything touching the database.
- `AddTransient` — new instance every time it's asked for. Use for very lightweight services.
- `AddSingleton` — one instance for the whole app. Use for caches, config readers, etc.

### Database Transactions

A transaction guarantees that a group of database operations either ALL succeed or ALL fail:

```csharp
using var transaction = await _db.Database.BeginTransactionAsync();
try
{
    // do multiple db operations...
    book.Stock -= item.Quantity;
    _db.Orders.Add(order);
    await _db.SaveChangesAsync();

    await transaction.CommitAsync();  // makes all changes permanent
}
catch
{
    await transaction.RollbackAsync(); // reverts ALL changes if anything failed
    throw;
}
```

### DTOs vs Models

| | Model | DTO |
|-|-------|-----|
| Lives in | `Models/` | `DTOs/` |
| Maps to | Database table | API request/response |
| Contains | Navigation properties, all columns | Only what the API needs |
| Used by | EF Core | Controllers / Services |

**Rule:** never return a Model class directly from a controller endpoint. Always map to a DTO.

### Nullable Reference Types (`= null!`)

When you see `public string Name { get; set; } = null!;` — the `null!` tells the compiler
"I know this will be set by EF Core/model binding, trust me". This avoids false nullable warnings.

## How to Adapt This for a Test

When given a test problem, the steps are always:

1. **Read the task** — identify what entities (tables) exist and how they relate
2. **Create Models** — one C# class per table in `Models/`
3. **Create Configurations** — one `IEntityTypeConfiguration<T>` per model in `Configurations/`
4. **Update AppDbContext** — add `DbSet<T>` for each model
5. **Create DTOs** — what does each endpoint return? One DTO class per shape
6. **Create Migrations** — `dotnet ef migrations add InitialCreate`
7. **Apply Migrations** — `dotnet ef database update`
8. **Write Services** — LINQ queries go here
9. **Write Controllers** — thin, just call services and handle exceptions

### Minimal 5-minute checklist

```
Models/         ← C# class + attributes ([Key], [ForeignKey], [MaxLength], etc.)
Configurations/ ← HasKey, HasData (seed), HasOne/HasMany relationships
Data/           ← add DbSet<T> for each model
DTOs/           ← shape of request/response JSON
Services/       ← LINQ queries (Where, Select, Include, FirstOrDefaultAsync)
Controllers/    ← HttpGet/Post/Put/Delete + try/catch for NotFound, Conflict
Program.cs      ← AddDbContext + AddScoped for each service
appsettings.json← connection string
```

### Common mistakes to avoid

- Forgetting to call `await _db.SaveChangesAsync()` after adding/updating/removing
- Not wrapping multi-step operations in a transaction
- Returning a Model class directly from a controller (use DTOs!)
- Forgetting to register a service in `Program.cs` (will get `InvalidOperationException`)
- Forgetting `[FromBody]` on POST/PUT parameters (body won't be deserialized)
- Using `.Result` or `.Wait()` instead of `await` (causes deadlocks)
